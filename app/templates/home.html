The exercise in 60 seconds

You inherit a deliberately bad chatbot and a fully working measurement rig. Your job is to make the chatbot better — add personas, history, voice, better prompts — while watching exactly what each change costs in tokens, time, and compute on the Metrics tab.

The point isn't the chatbot. The point is learning to feel what a 200-token system prompt does to your context budget across a 20-turn session, and to measure it.

{# ── Five steps ──────────────────────────────────────────────────────── #}
1

Open Codespaces

Click Code → Codespaces → Create on the repo. Working dev environment in 30 seconds. No local setup.

2

Pick a feature

Choose from the Feature Backlog. Start with something tagged ★ easy.

3

Branch + build

Create a branch, make your change, run ./scripts/preflight.sh to catch issues before pushing.

4

Open a PR

Push and open a pull request. 9 automated agents review your code in parallel (~90 seconds). Fix what they find.

5

Merge → live

When all agents are green, merge. Your change deploys to this site in ~90 seconds. Watch the metrics move.

{# ── Getting started ─────────────────────────────────────────────────── #}

Getting started

First time? Walk through Your First PR — a step-by-step guide to opening a trivial PR that calibrates you to the tools, the agents, and the deploy loop. Takes ~5 minutes.

Three commands to remember:

./scripts/preflight.sh        # run all 9 agents locally before you push
./scripts/explain-failure.sh  # decode any failing agent check
rg <name>                     # fast search across the codebase

Need help? The AI Guide tab has a coding mentor that teaches concepts without writing your code for you. Ask it about Python, FastAPI, HTMX, the metrics tab, or any agent failure.

{# ── What the metrics tab shows ──────────────────────────────────────── #}

What the metrics tab shows you

Every chat turn is decomposed into the labelled slots that fed it: system prompt, persona, few-shot examples, conversation history, and the user message. Each slot's token count is tracked separately so you can see exactly where your tokens are going.

The key insight: every metric has an in/out or pre/post axis, and the asymmetry between the two halves is where the lessons live. Input tokens vs output tokens. Prefill time vs decode time. Network bytes in vs out. The Bidirectional Framing wiki page explains this in depth.

{# ── The class arc ───────────────────────────────────────────────────── #}

The 2.5-hour arc

0:00 – 0:20Orientation. Tour the site, demo the chat, demo the metrics tab, read one agent's prompt.
0:20 – 0:35Everyone opens a trivial first PR. Watch the agents run. Read every comment. Calibrate.
0:35 – 1:50Feature work. Pick from the backlog above. Open PRs, read agent feedback, iterate, merge, watch the deploy.
1:50 – 2:15Red-team round. Try to break someone else's merged feature. Bugs become the next PR backlog.
2:15 – 2:30Wrap. Look at the cumulative metrics together. Discuss which features cost more than they were worth.

Keep the Metrics tab open in a second browser tab the entire time. When you merge, watch what moves. That's the whole point.
