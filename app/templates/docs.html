Getting Started

You need: VS Code, Python 3.12+, git, and a GitHub account that's been added as a collaborator on the repo.

1. Clone the repo

git clone git@github.com:wize73-ai/slim.git
cd slim

If you haven't set up SSH keys with GitHub, use HTTPS instead:
git clone https://github.com/wize73-ai/slim.git

2. Create a virtual environment and install dependencies

# Create the venv
python3 -m venv .venv

# Activate it
# Mac/Linux:
source .venv/bin/activate
# Windows (PowerShell):
.venv\Scripts\Activate.ps1

# Install everything
pip install -r requirements.txt -r requirements-dev.txt

If python3 isn't found, try python. You need 3.12 or higher — check with python3 --version.

3. Open in VS Code

code .

VS Code will prompt you to install recommended extensions — click Install All. This gives you:

  • Ruff — auto-formats and auto-fixes Python on every save
  • Pylance — type checking, autocomplete, go-to-definition
  • mypy — strict type checking on core/
  • GitLens — git blame, history, PR integration
  • TOML + YAML + Jinja — syntax highlighting for config files and templates

The project ships .vscode/settings.json that configures ruff-on-save automatically. You don't need to change any VS Code settings.

4. Run the app locally

The chatbot needs an inference server. Since you can't reach the real one from your laptop, we ship a mock that returns canned responses — enough to test the UI, the metrics tab, and the projection calculator.

# Terminal 1: start the mock inference server
python .devcontainer/mock-guapo.py &

# Terminal 2: start the app
OPENAI_BASE_URL=http://127.0.0.1:9999/v1 uvicorn app.main:app --reload --port 8080

Open http://localhost:8080 in your browser. You should see the landing page. Click Chat and send a message — you'll get a mock response. Click Metrics — the token flow chart will populate with real (mocked) data.

The --reload flag makes uvicorn restart when you save a file, so you see changes instantly.

5. Verify everything works

# Run the preflight checks locally
./scripts/preflight.sh

If this prints ✓ preflight passed, you're ready to start building.

The Exercise

What you're building

You inherit a deliberately bad chatbot — single turn, no memory, no persona, no style. You also inherit a fully working measurement rig that shows you exactly what every chat turn costs in tokens, time, and compute.

Your job: make the chatbot better while watching the measurements. Add personas, conversation history, few-shot examples, streaming, voice, better UI — each as a separate PR. After each merge, watch the Metrics tab and see what your change actually cost.

The point isn't the chatbot. The point is learning to feel what design decisions cost.

The loop

  1. Branch from main: git checkout -b feature/add-cooking-persona
  2. Edit files under app/ (that's where your code goes)
  3. Preflight: ./scripts/preflight.sh runs all 9 agents locally in ~15 seconds
  4. Commit with a conventional message: feat: add cooking persona
  5. Push and open a PR
  6. 9 agents review your code in parallel (~90 seconds). Fix what they find.
  7. Merge when all agents are green
  8. Deploy happens automatically — your change is live at class.wize73.com in ~90 seconds
  9. Verify on the Metrics tab — watch what moved

Where you can edit

  • You CAN edit: anything under app/ — personas, system prompts, few-shot examples, templates, the chat handler, routes, static assets.
  • You CANNOT edit: core/, docker/, .github/, docs/, .devcontainer/. These are locked. If you need a change there, open an issue.
  • You CANNOT add dependencies. requirements.txt is locked. Need a new package? File an issue tagged needs-dep.

Commit message format

We use Conventional Commits. Agent 6 enforces this.

feat: add cooking persona with weeknight-dinner focus
fix: chat UI crashes when persona file is empty
docs: explain the token flow chart
refactor: split persona loader into its own module
test: add round-trip test for the security preamble

The 2.5-hour class arc

0:00 – 0:20Orientation. Tour the site, demo the chat, demo the metrics tab.
0:20 – 0:35Everyone opens a trivial first PR. Watch the agents. Calibrate.
0:35 – 1:50Feature work from the backlog below. Open PRs, iterate with agent feedback, merge, watch deploys.
1:50 – 2:15Red-team round. Try to break someone else's feature. Bugs become PRs.
2:15 – 2:30Wrap. Look at the cumulative metrics. Discuss what cost more than it was worth.

Your First PR

Walk this once before tackling a real feature. It calibrates you to the agents and the deploy loop.

  1. Branch: git checkout -b feature/my-first-persona
  2. Create a file at app/personas/cooking.yaml: 
    name: Cooking Assistant
description: A friendly home cook who explains techniques.
tone: warm, practical
constraints:
  - Prefer metric with imperial in parentheses
  - Never suggest raw eggs or raw pork
  3. Preflight: ./scripts/preflight.sh — should pass
  4. Commit: git add app/personas/cooking.yaml && git commit -m "feat: add cooking persona"
  5. Push: git push -u origin feature/my-first-persona
  6. Open the PR on GitHub (or gh pr create --fill). Fill in the template — especially the "AI assistant" and "cost impact" sections.
  7. Wait ~90 seconds for the 9 agents to run. All should pass for a file-only change.
  8. Merge when green.
  9. Watch the deploy in the GitHub Actions tab (~90 seconds).
  10. Verify at class.wize73.com. The persona file is now on disk in the running container.

Total time: ~5 minutes from branch to live. Real features have more iteration with the agents, but the shape is the same.

Architecture

Where things run

Three components, one host (for this class):

browser (you) ──→ Cloudflare (TLS + SSO) ──→ Cloudflare Tunnel
                                                    │
                                              ┌─────▼──────┐
                                              │   guapo     │
                                              │ ├ app       │  ← the chatbot (what you build)
                                              │ ├ sidecar   │  ← host metrics
                                              │ ├ agent-proxy│  ← CI's inference path
                                              │ ├ phi-4-mini │  ← chat (vLLM)
                                              │ ├ whisper    │  ← speech-to-text
                                              │ └ XTTS-v2   │  ← text-to-speech
                                              └─────────────┘

The chatbot and the AI models run on the same machine. Students never access the machine directly — they interact via the GitHub repo and the public URL.

The locked core

core/ is CODEOWNERS-protected. You consume its APIs but cannot edit it. This guarantees:

  • Measurement integrity — the token counts and timing numbers stay trustworthy across PRs. If students could tweak the measurement code, two PRs could report different costs for identical changes.
  • Security boundary — the security preamble (tells the model not to leak infrastructure details) and the output filter (catches leaks anyway) cannot be bypassed.
  • Dependency lock — you can't add Python packages without an instructor issue. Closes the supply-chain attack vector.

The Metrics Tab

Open /metrics in a second browser tab and keep it open during the entire class. When you merge a change, watch what moves.

Labelled slots

Every chat turn is built from labelled slots:

[security preamble — locked, ~280 tok]
[baseline prompt — locked, ~190 tok]
[your system prompt — initially empty]
[your persona — initially empty]
[few-shot examples — initially empty]
[conversation history — grows each turn]
[user message — the current input]

The metrics tab decomposes each turn into these slots so you can see exactly where the tokens come from. When you add a 60-token persona, you'll see the system slot grow by 60 — and that 60 is paid on every turn for the rest of the session.

Bidirectional framing

Every metric has an in/out or pre/post axis. The asymmetry between the two halves is where the lessons live:

  • Input vs output tokens — are you sending more than you're getting back, or vice versa?
  • Prefill vs decode time — prefill scales with input size, decode with output. Two different optimization knobs.
  • Network in vs out — catches "you're shipping the entire history to the model every turn"

A persona that summarises (short output from long input) clusters in a different place on the asymmetry chart than a persona that writes creatively (long output from short input). Watch where yours falls.

Projection calculator

On the metrics tab, enter a planned architecture (system size, persona size, few-shot count, history retention, planned turns) and see projected per-turn costs before you build anything. Compare the projection to your actual session afterwards — the gap is where the learning happens.

The 9 Agents

Every PR runs these 9 agents in parallel. All must pass for the merge button to enable. Run ./scripts/preflight.sh locally to catch most issues before pushing.

If an agent fails and you don't understand why: ./scripts/explain-failure.sh <agent-name>

1. secrets-scan

Catches hardcoded secrets, leaked URLs, anything that exposes the inference server's identity.

Fix: never hardcode URLs or IPs. Use os.environ['OPENAI_BASE_URL']. Don't put server names in comments either.

2. code-quality

ruff format + ruff check (strict) + mypy strict on core/ + pytest + docstring coverage. Auto-fixes what it can and commits the fixes back to your branch.

Fix: ruff format . && ruff check --fix . && mypy core/ && pytest. Most failures auto-resolve. The ones that don't need type hints or docstrings added manually.

3. red-team

Builds your container and runs 12 adversarial probes — tries to extract the server URL, leak the system prompt, jailbreak the persona. Also checks a usability canary (your chatbot must NOT refuse legitimate questions).

Fix: don't override the security preamble. Don't make your persona so restrictive it refuses everything. The security preamble in core/chat/security.py already handles refusals for infrastructure questions.

4. blue-team

Defensive code review. Checks that calls to the inference server are wrapped in try/except, that user input is validated, that error handling exists.

Fix: wrap stream_completion() calls with except UpstreamUnavailable / except UpstreamTimeout / except ChatError. Return a friendly error banner, not a crash.

5. classroom-safety

Reviews the content of new persona / system prompt / few-shot files for age-appropriateness. Different from secrets-scan — this is about WHAT the chatbot is told to say.

Fix: don't put profanity in persona text. Don't impersonate real teachers or students by name.

6. version-discipline

Pinned dependencies, locked paths untouched, conventional commit messages, PR description non-empty.

Fix: use feat: / fix: / docs: commit prefixes. Don't edit files under core/. Fill in the PR description template.

7. build-and-smoke

Builds the Docker image from your branch and runs a smoke test. Verifies the metrics tab's locked markers are still present — catches PRs that accidentally break the measurement surface.

Fix: if the build fails, you have a Python syntax error. Run python3 -c "import ast; ast.parse(open('your_file.py').read())". If the smoke test fails, you broke a locked template — revert that change.

8. slop-and-scope

AST-based detection of dead code, premature abstractions, hallucinated imports, and scope creep (your diff vs the linked issue).

Fix: delete functions nothing calls. Remove abstractions with only one consumer. Verify every from X import Y points at a real module. Stick to the issue scope — if you found something else to fix, open a separate PR.

9. malicious-code-review

Insider-threat detection. Semgrep + bandit + AST scanner for danger primitives (eval, exec, subprocess shell=True) + entropy detector for obfuscated payloads + time-bomb pattern detector. Intentionally paranoid — false positives are cheap.

Fix: if you tripped this legitimately, explain what you were doing in the PR description. The instructor has been notified and will review. Most common false positives: dynamic string construction, env var reads with TOKEN/KEY in the name.

Feature Backlog

Pick one. Each is sized for a single PR in one sitting. The full list with details is at the wiki.

★ easy (~15 min)

  • Add a persona — one YAML file under app/personas/
  • Add a few-shot example block — YAML under app/examples/
  • Improve the chat UI — edit app/templates/chat.html. Typography, layout, anything.

★★ medium (~30-60 min)

  • Wire up persona loading — make build_request() actually USE persona files
  • Add conversation history — session state + the history slot. Watch the trajectory chart ramp.
  • Add a temperature slider — control the model's randomness per turn
  • Add SSE streaming — tokens appear as they arrive. HTMX sse-swap.
  • Improve error banners — different messages for different failures

★★★ hard (~1-2 hours)

  • Voice input — microphone → whisper STT → chat. The endpoint is on the backend already.
  • Voice output — XTTS-v2 TTS → audio playback. Same backend.
  • Persona picker dropdown — read app/personas/, show a dropdown, track per-persona metrics
  • Content filter — validate user input before sending to the model
  • Token budget warnings — show a banner when context fills up
  • Compare two prompts side by side — the pedagogical power move

Troubleshooting

An agent is failing and I don't understand why

./scripts/explain-failure.sh          # explains current PR's failures
./scripts/explain-failure.sh red-team # explains a specific agent
./scripts/explain-failure.sh --list   # lists all 9 agents

My changes work locally but the deploy fails

The deploy auto-rolls back within 30 seconds. Check the GitHub Actions tab for the failing step. Most common: a startup error in your code. The instructor can check docker logs on the server.

The public URL returns 502

The app container is down. The instructor will restart it. Wait 30 seconds and refresh.

The metrics tab is blank

No chat turns have happened yet — the buffer starts empty. Have a conversation and the charts populate.

I want to roll back a broken merge

The auto-rollback handles most broken deploys within 30 seconds. For manual rollback, the instructor has a one-command script on the server. Don't panic — fix forward in the next PR.

I broke main

The deploy auto-rolled back (if it tripped the health check) or the instructor will roll back manually. Open a follow-up PR that fixes the issue. Don't try git revert — just fix forward.

I can't push because branch protection rejects me

You're trying to push directly to main. Open a PR instead — direct pushes are blocked.

Docker build fails on my Mac (ARM)

docker buildx build --platform=linux/amd64 -f docker/app/Dockerfile -t wize73-app:local .

I have no idea what I'm doing

Walk through Your First PR above, step by step. By the end you'll have made one successful PR and understand the full loop. Then ask the AI Guide mentor any concept questions.

None of this helps

Tag the instructor in your PR description or open an issue. They're monitoring during class and will respond quickly.
