AI Guide

How to use AI assistants ethically for velocity coding, plus a mentor chatbot that teaches without doing.

{# ── The velocity coding manifesto ───────────────────────────────────── #}

The velocity coding contract

You are responsible for what you merge. Regardless of who or what wrote the first draft. When an agent flags a problem, "the AI wrote it" is not a defense — it's the diagnosis.
AI is the accelerator, not the author. The best velocity-coding workflow is: think about what you need → have the AI draft it → read and understand every line → adapt it to the project → verify with preflight → commit. The "read and understand" step is the one people skip and regret.
Attribution is mandatory, not shameful. The PR template asks which AI tool you used and what you asked it. This isn't a gotcha. Instructors use the data to spot patterns across the class. Students who write "asked Claude to add a persona and copy-pasted the result" aren't in trouble — they're getting the most useful kind of feedback.
The agents are your AI's editor. Nine automated agents review every PR. They catch the slop patterns AI tools produce (unused imports, hallucinated modules, premature abstractions, dead code, missing error handling). Think of them as a code review from someone who's seen every AI mistake before.
{# ── The velocity loop ───────────────────────────────────────────────── #}

The velocity loop

1
Think — understand what you're building and why. Read the issue. Read the relevant wiki page. Know what metric should move when this ships.
2
Prompt — ask your AI assistant for a draft. Be specific: "write a Python function that loads a YAML persona file and returns the text" not "add personas to the chatbot."
3
Read and understand every line. This is the step that separates velocity coding from copy-paste. If you can't explain what a line does, don't commit it. Ask the mentor below if something is unclear.
4
Adapt — the AI doesn't know this project's conventions. Does it use build_request() with the labelled slots? Does it wrap guapo calls in try/except? Does it follow the ruff config? Adapt the draft to fit.
5
Preflight — ./scripts/preflight.sh runs all 9 agents locally in ~15 seconds. Fix what it catches before pushing. This is the fastest feedback loop in the whole class.
6
Commit + push — conventional commit format. Open a PR. The 9 agents run for real. Iterate until green. Merge. Watch the metrics.
{# ── Common AI slop patterns ─────────────────────────────────────────── #}

AI slop to watch for

These are the patterns the agents catch most often in AI-generated code. Learn to spot them yourself and you'll save time.

Unused importsAI adds import os, sys, json speculatively. If you don't use it, delete it. ruff F401 catches this.
Hallucinated modulesAI invents from fastapi.utils import magic_helper that doesn't exist. The slop agent catches this via importlib.util.find_spec.
Dead helper functionsAI writes a utility function "for later" that nothing calls. If nothing calls it, delete it.
Premature abstractionAI creates BasePersonaLoader with one subclass. You don't need a base class for one thing. Just write the one thing.
Broad exceptionsexcept Exception: pass swallows everything including bugs you'd want to see. Catch specific exceptions. ruff TRY catches this.
Missing type hintsAI often omits return types and parameter types. mypy strict catches this on core/. Add them — they're documentation that compilers read.
Redundant comments# increment x above x += 1 is noise. Comments should explain WHY, not WHAT.
Scope creep"Add a persona" turns into a 400-line refactor. Stick to the issue scope. If you find something else to fix, open a separate PR.
{# ── When to ask for help vs. when to figure it out ──────────────────── #}

When to ask for help

Ask the mentor below when you need to understand a concept, decode an error, or review your approach before committing.

Ask your AI assistant when you need a code draft, a boilerplate template, or a syntax reference.

Ask the instructor when an agent failure doesn't make sense after you've tried the mentor and ./scripts/explain-failure.sh, or when you want to change something in the locked core.
