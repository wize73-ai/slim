# `core/python/` — pointer to the locked dependency manifest

The actual dependency files live at the **repository root**, where standard
Python tooling (ruff, mypy, pytest, pip, uv) expects them:

| File | What it is |
|---|---|
| [`/pyproject.toml`](../../pyproject.toml) | Project metadata, dependency declarations, ruff/mypy/pytest/interrogate/bandit configuration. **Source of truth.** |
| [`/requirements.txt`](../../requirements.txt) | Production deps mirrored from pyproject for tools that don't read TOML. |
| [`/requirements-dev.txt`](../../requirements-dev.txt) | Dev/test deps (ruff, mypy, pytest, bandit, semgrep — the tools the 9 PR agents use). |
| `/uv.lock` or `/requirements-lock.txt` | The full transitive dependency graph with hash verification. Generated by `uv pip compile`. Will be added in a follow-up commit when the devcontainer is built. |

This directory exists for navigability — students browsing `core/` will look
here for the dependency story and the README explains where to find it.

## Why locked

**Students cannot add dependencies.** All three files at the root are
CODEOWNERS-protected. The supply-chain attack vector — typo-squat packages,
malicious git URLs, CVE-ridden libraries — is closed at the source.

If you genuinely need a new package:

1. Open an issue tagged `needs-dep`
2. Justify why you need it (and why an existing dep can't do the job)
3. An instructor reviews, adds it to `pyproject.toml`, regenerates the lockfile,
   opens a PR, and merges after the agents pass
4. You rebase and pull in the new dep

## Why pinned

Two reasons:

1. **Reproducibility.** Two students running the same commit get bit-identical
   environments. Without pinning, "works on my machine" bugs come from
   transitive dependency drift.
2. **Security.** Pinned versions can be audited for CVEs. Floating versions
   can introduce known vulnerabilities at any time.

## Regenerating the manifest

When updating versions (instructor only):

```bash
uv pip compile pyproject.toml -o requirements.txt --upgrade
uv pip compile pyproject.toml --extra dev -o requirements-dev.txt --upgrade
```

Commit the result. The PR agents will run on it like any other change.

## Tooling configuration also lives in pyproject.toml

The `[tool.ruff]`, `[tool.mypy]`, `[tool.pytest.ini_options]`,
`[tool.interrogate]`, and `[tool.bandit]` sections in `pyproject.toml` are
the configuration the PR agents use. Read those sections to understand
exactly what `agent 2 (code-quality)` is checking on every PR.
